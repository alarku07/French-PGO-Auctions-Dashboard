## Summary

Add Docker container images for the FastAPI backend and Vue.js frontend, a dedicated nginx reverse proxy, and Docker Compose orchestration for both development (hot-reload) and production-like deployment. A Makefile provides commands for building, tagging, and pushing images to GitHub Container Registry. Database migrations run automatically via the backend entrypoint before the API accepts requests.

---

## Technical Context

**Language/Version**: Python 3.14 (backend), Node.js 24 (frontend build), nginx 1.29.5
**Primary Dependencies**: FastAPI + Uvicorn, SQLAlchemy 2.0 async, asyncpg, Alembic, structlog (backend); Vue 3 + Vite 6, TypeScript 5 (frontend); nginx:1.29.5-alpine (proxy)
**Storage**: PostgreSQL 16 (via `postgres:16-alpine` image); named Docker volume for persistence
**Testing**: pytest + pytest-asyncio (backend); Vitest (frontend) ‚Äî run on host, not inside Docker
**Target Platform**: Linux containers (Docker Engine v24+ / Docker Desktop)
**Project Type**: Web application ‚Äî Python/FastAPI backend + Vue.js/Vite frontend
**Performance Goals**: Stack startup < 90s; hot-reload reflect < 5s; image builds < 5 min
**Constraints**: Single host port (80) via reverse proxy in production; restart on-failure policy; stdout-only logging; no venv inside Docker images (constitution exception documented in research.md)
**Scale/Scope**: Single-host deployment; 4-service compose stack (proxy, frontend, backend, postgres)

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-checked after Phase 1 design.*

### I. Data Integrity ‚úÖ PASS

- PostgreSQL is used as the database in all compose configurations.
- Alembic `upgrade head` runs automatically on startup ‚Äî only additive migrations; no `DROP`/`TRUNCATE` without explicit approval.
- The `postgres_data` volume ensures data survives container restarts.
- SQLite is excluded from all Docker configurations.

### II. Quality Gates ‚úÖ PASS

- Linting (`ruff check`), type checking (`mypy`), and unit tests (`pytest`) are run on the host before building images ‚Äî Docker build does not bypass them.
- The `Makefile` `ci` target runs all three gates before `build`.
- ESLint + vue-tsc run for the frontend before build.

### III. Python Discipline ‚öÝÔ∏è DOCUMENTED EXCEPTION ‚Äî PASS WITH NOTE

- **Exception**: Virtual environments are not used inside Docker images. `pip install .` installs packages globally into the container.
- **Justification**: Docker containers are themselves isolation boundaries. The constitution explicitly permits skipping venvs when "a venv cannot be used (e.g., inside a Docker image build)" provided the decision is documented. See [research.md ‚Äî RES-003](./research.md).
- **Local development unchanged**: `.venv/` continues to be used for all host-side development.

### IV. Frontend Standards ‚úÖ PASS

- Vue.js Composition API, TypeScript, and responsive layout are unchanged.
- The Docker build runs `vue-tsc --noEmit && vite build` ‚Äî the same build that enforces type safety.

### V. Observability ‚úÖ PASS

- All containers write to stdout/stderr only.
- The backend's existing `print(f"[PGO-API] running on port {settings.port} (PID: {pid})")` startup line is preserved.
- structlog structured logging for every endpoint is already implemented.

## Implementation Phases

### Phase A ‚Äî Backend Container

1. Create `src/backend/.dockerignore`
2. Create `src/backend/Dockerfile` (multi-stage: builder ‚Üí runtime)
   - Builder: `python:3.14-slim`, `apt install build-essential postgresql-client`, `pip install .`
   - Runtime: `python:3.14-slim`, non-root user UID 10001, copy site-packages + app source
   - Build args: `VERSION`, `BUILD_DATE`
3. Create `src/backend/docker-entrypoint.sh`
   - `pg_isready` wait loop (uses `POSTGRES_HOST`, `POSTGRES_PORT`, `POSTGRES_USER`)
   - `alembic upgrade head` (exits non-zero on failure)
   - `python -m app.main`
4. Add `GET /health` endpoint to `src/backend/app/main.py`

### Phase B ‚Äî Frontend Container

1. Create `src/frontend/.dockerignore`
2. Create `src/frontend/Dockerfile` (multi-stage: node builder ‚Üí nginx runtime)
   - Builder: `node:24-slim`, `npm ci`, `npm run build`
   - Runtime: `nginx:1.29.5-alpine`, copy `dist/` to `/usr/share/nginx/html`
   - Include SPA nginx config (`try_files`)
   - Build args: `VERSION`
3. Update `src/frontend/vite.config.ts` ‚Äî `server.host: true` (dev Docker access)

### Phase C ‚Äî Reverse Proxy

1. Create `docker/nginx/nginx.conf` (production)
   - Upstream blocks: `backend:8000`, `frontend:80`
   - `/api/*` ‚Üí backend; `/*` ‚Üí frontend
   - Internal `/health` location
2. Create `docker/nginx/nginx.dev.conf` (development)
   - Same routing as production
   - Additional WebSocket proxy for Vite HMR (`Upgrade` header support)
   - Frontend upstream: `frontend:5173`

### Phase D ‚Äî Compose Orchestration

1. Create `docker-compose.yml` (production-like)
   - Services: `nginx-proxy`, `frontend`, `backend`, `postgres`
   - Only `nginx-proxy` exposes host port 80
   - `postgres` health check with `pg_isready`
   - `backend` health check with `curl /health`
   - `depends_on` with `condition: service_healthy`
   - `restart: on-failure`
   - `postgres_data` named volume
2. Create `docker-compose.dev.yml` (dev overrides)
   - `backend` override: build target dev (or source mount `./src/backend:/app`)
   - `frontend` override: build target dev, source mount `./src/frontend:/app` (excl. node_modules)
   - `nginx-proxy` override: use `nginx.dev.conf`
3. Create `.env.docker.example` with all variables and placeholder values

### Phase E ‚Äî Makefile & Registry

1. Create `Makefile` at project root with targets:
   - `make dev` ‚Äî start dev stack
   - `make up` ‚Äî start production stack
   - `make down` ‚Äî stop stack
   - `make build [SERVICE=backend|frontend]` ‚Äî build images with VERSION tag
   - `make push [SERVICE=...] [VERSION=...]` ‚Äî push to GHCR
   - `make ci` ‚Äî run lint + type check + tests (host), then build images

## Complexity Tracking

| Constitution Note | Why | Simpler Alternative Rejected Because |
|-------------------|-----|--------------------------------------|
| No venv in Docker image | Docker container is the isolation boundary; pip installs globally into container layer | A venv inside Docker adds complexity, wastes space, and provides no isolation benefit that the container itself doesn't already provide |
